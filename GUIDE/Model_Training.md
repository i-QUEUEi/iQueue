### iQueue ML Pipeline | Full Walkthrough

## WHAT IS THIS SECTION?

`src/model_implementation/` and `src/Evaluation/` together form the
**training and evaluation layer** of iQueue.

They take the preprocessed DataFrame and do six things:

1. Audit the raw data for quality problems (before training)
2. Split data into train/test sets (two different ways)
3. Train all 3 models on the data
4. Evaluate each model three different ways
5. Pick the winner (lowest robust_mae)
6. Save outputs: model file, charts, report

**They do NOT generate or preprocess data.** That is `Preprocessing/`'s job.
**They do NOT make predictions.** That is `Prediction/`'s job.

## THE RANDOM SEED

```python
RANDOM_STATE = 42
```

Passed to every model and every split. Ensures identical results every run.
If you change this number, the random 80/20 split changes, which slightly
changes the data each model sees, which changes the rankings slightly.

## STEP 1 — DATA QUALITY AUDIT (evaluate_data_quality)

Before any model sees the data, `evaluate_data_quality()` reads the
**raw CSV** (not the preprocessed version) and computes 13 statistics:

```python
summary = evaluate_data_quality(raw_df)
```

**What it checks:**

| Statistic | What it catches |
|---|---|
| rows | Total transaction count |
| columns | Column count (sanity check) |
| duplicate_rows | Exact repeated records |
| missing_cells | Any blank values in any column |
| negative_waiting_rows | Impossible wait times (< 0) |
| negative_queue_rows | Impossible queue lengths (< 0) |
| target_mean | Average wait time across all records |
| target_median | Middle wait time (less affected by extremes) |
| target_std | How spread out wait times are |
| target_min / target_max | Shortest and longest waits recorded |
| target_p10 / target_p90 | 10th and 90th percentile waits |

**Why check the raw CSV, not the preprocessed one?**

The raw CSV is what the user actually controls. If it has duplicates
or nulls, we want to know before spending time training. The
preprocessed CSV will already have bad rows removed — auditing it
would hide the original problems.

**Current dataset results:**
```
Rows: 21,428  |  Duplicates: 0  |  Missing: 0
Wait time — Mean: 36.6 min, Std: 22.0 min, Max: 90.0 min
```

## STEP 3 — TWO SPLITS

Training creates **two completely different splits** of the same data.
Each split is used to evaluate the model a different way.

### Split 1 — Random 80/20

```python
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, random_state=RANDOM_STATE
)
```

- 80% of rows → training (17,142 rows)
- 20% of rows → testing (4,286 rows)
- Rows are chosen **randomly** — a Monday from March could land in test,
  a Friday from September in train

**What it tests:** Overall accuracy on unseen data.

### Split 2 — Chronological (splits.py)

```python
chrono_train, chrono_test, train_dates, test_dates = chronological_split(df, features)
```

- Oldest 80% of **unique dates** → training (176 dates, roughly Jan–Aug)
- Newest 20% of unique dates → testing (44 dates, roughly Sep–Nov)
- Splits by **date**, not rows — a busy Monday with 120 rows counts as one date,
  just like a quiet Wednesday with 80 rows

**What it tests:** Can the model predict FUTURE dates it has never seen?

**Why date-based, not row-based?**

If we split row-by-row, Tuesday Sep 5 might have some rows in train
and some in test. The model would effectively "see the future" during
training. Splitting by date prevents this leakage.

**Before split:**
```
Jan 1 → Jan 2 → ... → Aug 31 → Sep 1 → ... → Nov 3
|←────── train (176 dates) ────────→|←── test (44 dates) ──→|
```

## THE 3 MODELS

### Why compare 3 models?

No single model is universally best. By benchmarking all three,
we find which one actually fits THIS data best, not which one is
theoretically best in general.

### Model 1 — LinearRegression

```python
# Formula: ŷ = β₀ + β₁x₁ + β₂x₂ + ... + β₁₆x₁₆
LinearRegression()   # no hyperparameters needed
```

- Fits a straight line (hyperplane) through the data
- Learns one coefficient (β) per feature — how much each feature shifts wait time
- **Why it's included:** Simplest possible model. If wait times are mostly linear,
  it should perform comparably to more complex models. It also shows us
  which features have the largest linear effect.
- **Its limit:** Can't capture nonlinear interactions (e.g., "peak hour AND peak day"
  together is worse than their individual effects suggest)

**What the training output shows:**
```
Intercept (β₀): -0.3243
Top coefficients:
  is_peak_day    β = +6.19  ↑  (being Mon/Fri adds 6 min)
  is_peak_hour   β = +3.97  ↑  (peak hour adds 4 min)
  is_holiday     β = -1.85  ↓  (holidays reduce wait by 2 min)
```

### Model 2 — RandomForest

```python
# Formula: ŷ = average of 500 decision trees
RandomForestRegressor(
    n_estimators=500,     # 500 trees
    max_depth=15,         # max 15 levels deep per tree
    min_samples_split=5,  # need 5 samples to create a branch
    min_samples_leaf=2,   # each leaf must cover ≥2 data points
    max_features="sqrt",  # each tree sees √16 ≈ 4 random features
)
```

- Builds 500 trees **independently and in parallel**
- Each tree trains on a random subset of rows AND a random subset of features
- Final prediction = average of all 500 trees
- **Why it's included:** Handles nonlinear relationships and feature interactions
  naturally. No scaling needed. Resistant to noisy features.
- **Its limit:** Can overfit if max_depth is too high (solved by capping at 15)

**What the training output shows:**
```
Trees: 500  |  Max depth: 15  |  Actual depths: min=15, avg=15.0, max=15
Top feature importances:
  queue_length_at_arrival   0.2685  ██████████
  waiting_time_lag1         0.2656  ██████████
  service_time_min          0.1726  ██████
```

**Why max_features="sqrt"?**
Each tree only sees √16 ≈ 4 randomly selected features at each split.
This forces diversity — trees can't all latch onto the same dominant
features. Their predictions therefore capture different aspects of the data,
and averaging gives a more robust final answer.

### Model 3 — GradientBoosting

```python
# Formula: ŷ = F₀ + η·h₁(x) + η·h₂(x) + ... + η·h₂₅₀(x)
GradientBoostingRegressor(
    n_estimators=250,    # 250 sequential correction rounds
    learning_rate=0.05,  # each tree fixes only 5% of remaining error
    max_depth=3,         # deliberately shallow trees
    subsample=0.9,       # each tree trains on random 90% of rows
)
```

- Builds 250 trees **sequentially** — each corrects the mistakes of the previous
- `F₀` = initial guess (mean of all wait times)
- Each `η·hₙ(x)` = a shallow tree that reduces remaining error by 5%
- **Why it's included:** Can capture residual structure that RandomForest misses
  because it focuses each tree on the current errors specifically.
- **Its limit:** Slower to train, more sensitive to hyperparameters

**RandomForest vs GradientBoosting:**

| | RandomForest | GradientBoosting |
|---|---|---|
| Trees built | Independently (parallel) | Sequentially (one at a time) |
| Each tree focuses on | Random sample of data | The current errors |
| Tree depth | Deep (15) | Shallow (3) |
| Learning rate | N/A | 0.05 (conservative) |
| Speed | Faster | Slower |

## STEP 4 — 3-WAY EVALUATION (evaluate_model)

For each model, `evaluate_model()` runs **3 independent evaluations** and
combines them into one `robust_mae` score.

### Evaluation Way 1 — Random Split Test

```python
fitted_model.fit(X_train, y_train)     # train on 17,142 rows
test_pred = fitted_model.predict(X_test)  # predict 4,286 unseen rows
test_metrics = compute_metrics(y_test, test_pred)
```

Also predicts on training data to check for overfitting:
```python
train_pred = fitted_model.predict(X_train)
```
If `train_mae` is much lower than `test_mae`, the model memorized
training data instead of learning generalizable patterns.

### Evaluation Way 2 — Chronological Split Test

```python
chrono_model.fit(chrono_train[0], chrono_train[1])  # train on Jan–Aug
chrono_pred = chrono_model.predict(chrono_test[0])  # predict Sep–Nov
chrono_metrics = compute_metrics(chrono_test[1], chrono_pred)
```

This is a **separate model** trained only on old data.
A model can score well on random split by memorizing date patterns —
chronological split tests whether it actually generalizes to the future.

### Evaluation Way 3 — 5-Fold Cross-Validation

```python
cv = KFold(n_splits=5, shuffle=True, random_state=RANDOM_STATE)
cv_results = cross_validate(clone(model), X_train, y_train, cv=cv, ...)
cv_mae = -cv_results["test_mae"].mean()
```

**How 5-fold works:**
```
All training data (17,142 rows) split into 5 equal chunks:
│  Fold 1  │  Fold 2  │  Fold 3  │  Fold 4  │  Fold 5  │

Round 1: Train on [2,3,4,5] → Test on [1] → MAE₁
Round 2: Train on [1,3,4,5] → Test on [2] → MAE₂
Round 3: Train on [1,2,4,5] → Test on [3] → MAE₃
Round 4: Train on [1,2,3,5] → Test on [4] → MAE₄
Round 5: Train on [1,2,3,4] → Test on [5] → MAE₅

cv_mae = average(MAE₁ ... MAE₅)
```

**Why 5 different test sets?** One test set might happen to be easy or hard.
Averaging over 5 gives a more stable estimate of true performance.

**Note:** sklearn returns negative MAE (since it always maximizes scores).
The `-` sign flips it back to positive.

### Robust MAE — The Winner Criterion

```python
robust_mae = mean([test_metrics["mae"], chrono_metrics["mae"], cv_mae])
```

**A model must score well on ALL 3 methods to win.**

- Only good on random split? Might be overfitting.
- Only good on chronological? Might be underfitting (too simple).
- Only good on CV? Might not generalize to truly held-out data.

The lowest `robust_mae` wins — it must be consistently accurate across all evaluation scenarios.

## THE METRICS — metrics.py

```python
def compute_metrics(y_true, y_pred):
    return {
        "mae":  mean_absolute_error(y_true, y_pred),
        "rmse": sqrt(mean_squared_error(y_true, y_pred)),
        "r2":   r2_score(y_true, y_pred),
    }
```

### MAE — Mean Absolute Error

```
MAE = average of |actual - predicted| for all predictions
```

**Plain English:** On average, how many minutes off are we?

```
Prediction: 35 min   Actual: 32 min   Error: 3 min
Prediction: 60 min   Actual: 68 min   Error: 8 min
Prediction: 18 min   Actual: 19 min   Error: 1 min

MAE = (3 + 8 + 1) / 3 = 4.0 min
```

Best model achieves **MAE ≈ 2.9 min** — off by less than 3 minutes on average.

### RMSE — Root Mean Squared Error

```
RMSE = sqrt( average of (actual - predicted)² )
```

**Plain English:** Like MAE but large errors count more (they are squared first).
An error of 10 min counts as 100 in the average, not 10.

RMSE > MAE always. If they're very close, most errors are similar in size.
If RMSE >> MAE, there are occasional very large errors.

### R² — R-Squared

```
R² = 1 - (sum of squared model errors) / (sum of squared deviations from mean)
```

**Plain English:** What fraction of the variation in wait times does the model explain?

| R² | Meaning |
|---|---|
| 1.0 | Perfect — model explains all variation |
| 0.964 | Model explains 96.4% of variation (our RandomForest result) |
| 0.0 | Model is no better than just guessing the mean every time |
| < 0 | Model is worse than the mean (actively harmful) |

```
Baseline (always guess avg): MAE = 18.18 min  |  R² = -0.0002
```

### The Baseline

```python
baseline_value = y_train.mean()                          # e.g., 36.6 min
baseline_test_pred = np.full(y_test.shape, baseline_value)  # predict 36.6 for every row
baseline_metrics = compute_metrics(y_test, baseline_test_pred)
```

If we just predict "36.6 minutes" for every single transaction regardless of
day, hour, or queue — MAE = 18.18 min and R² ≈ 0.

**RandomForest beats baseline by 15.26 min MAE.** That 15-minute improvement
is the measurable value of having a trained ML model.

### Percentile Errors

Beyond averages, evaluation also checks the distribution of errors:

```python
abs_errors = |y_test - test_pred|
p90_abs_error = 90th percentile of abs_errors   # = 7.20 min
p95_abs_error = 95th percentile of abs_errors
max_abs_error = worst single prediction          # = 25.23 min
```

**What P90 = 7.20 min means:**
90% of predictions are within 7.20 minutes of the truth.
Only 10% of predictions are off by more than 7.20 minutes.

### Segment Error Analysis

Checks if the model is systematically worse on specific days or hours:

```python
# Is the model worse on Mondays than Wednesdays?
day_error = eval_df.groupby("day_name")["abs_error"].agg(["mean", "median", "max"])

# Is the model worse during peak hours?
peak_hour_error = eval_df.groupby("is_peak_hour")["abs_error"].mean()
peak_day_error  = eval_df.groupby("is_peak_day")["abs_error"].mean()
```

If peak_day MAE >> non-peak MAE, the model struggles with busy days —
a signal that more peak-day training data or features are needed.

### Feature Importance

```python
feature_importance = get_feature_importance(fitted_model, features, X_test, y_test)
```

Three strategies depending on model type:

**RandomForest / GradientBoosting:**
```python
values = model.feature_importances_
```
Each tree records which features it used most to reduce prediction error.
Averaged across all trees, this gives a stable importance ranking.

**LinearRegression:**
```python
values = np.abs(model.coef_)
```
Larger absolute coefficient = that feature shifts the prediction more.

**Other models (fallback):**
Permutation importance — shuffle one feature's values randomly and measure
how much worse the model gets. Big drop = that feature was important.

**Current ranking (RandomForest winner):**
```
queue_length_at_arrival   0.2685  (27% of predictive power)
waiting_time_lag1         0.2656  (27%)
service_time_min          0.1726  (17%)
queue_length_lag1         0.1114  (11%)
is_peak_day               0.0819  ( 8%)
```

The two lag features together account for ~38% of the model's predictive power —
confirming that knowing what happened to the previous person in the queue
is the single most powerful signal.

## WHY THESE 3 MODELS?

```
LinearRegression  — answers: is the relationship close to linear?
RandomForest      — answers: can nonlinear interactions improve accuracy?
GradientBoosting  — answers: can sequential error-correction do even better?
```

Each addresses a different hypothesis. The benchmark determines which
hypothesis holds for THIS dataset. The winner (RandomForest, R²=0.964)
confirms that nonlinear interactions matter and ensemble averaging works
better than sequential boosting for this particular data.

**Why no neural networks?**
- Dataset is small (~21K rows) — deep learning needs millions to shine
- All features are structured tabular data — tree models excel at this
- Interpretability matters (feature importances, coefficients) — neural
  networks are black boxes

**Why no feature scaling?**
Tree-based models (RandomForest, GradientBoosting) are completely
scale-insensitive — they only care about whether feature A > threshold,
not the actual value. Scaling would have no effect. Linear Regression
can benefit from scaling, but the features are already numeric and
comparably ranged, so results are acceptable without it.
